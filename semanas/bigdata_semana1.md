¿Por qué Big Data importa?

::image::

::title::
Big Data: idea central

::content::
Big Data se refiere al manejo, procesamiento y análisis de conjuntos de datos que superan las capacidades tradicionales de almacenamiento, consulta y análisis.

No se trata únicamente de “muchos datos”.

Implica nuevas arquitecturas, tecnologías, modelos de procesamiento, prácticas de gobierno y criterios éticos para transformar datos en valor.

<!--
Notas del presentador:
Aclarar que Big Data no se define solo por el tamaño del archivo o de la base de datos. Una organización puede tener muchos datos y no hacer Big Data si no cuenta con procesos de integración, análisis, escalabilidad y generación de valor. También puede iniciar con volúmenes moderados, pero requerir arquitecturas Big Data por la velocidad, variedad o complejidad del procesamiento. Resaltar que Big Data une infraestructura, analítica, negocio y responsabilidad social.
-->

::title::
Las dimensiones principales de Big Data

::left::
**Volumen**

Cantidad masiva de datos.

**Variedad**

Datos estructurados, semiestructurados y no estructurados.

**Velocidad**

Generación, ingesta y procesamiento continuo.

::right::
**Veracidad**

Calidad, confiabilidad y consistencia.

**Valor**

Capacidad de producir decisiones útiles.

**Vinculación ética**

Uso responsable, seguro y legítimo de la información.

<!--
Notas del presentador:
Desarrollar las dimensiones clásicas de Big Data con ejemplos. Volumen puede ser millones de transacciones o registros de sensores. Variedad incluye tablas, textos, imágenes, videos, audios, logs, JSON, XML y publicaciones en redes sociales. Velocidad aparece en streaming, pagos, monitoreo industrial o alertas de seguridad. Veracidad exige calidad y limpieza de datos. Valor implica que el análisis contribuya a una decisión. Añadir la dimensión ética como condición transversal: no todo dato disponible debe ser usado de cualquier manera.
-->

::title::
Tipos de datos en un ecosistema Big Data

::content::
**Estructurados:** datos organizados en tablas, filas y columnas.

**Semiestructurados:** datos con organización flexible, como JSON, XML o logs.

**No estructurados:** textos libres, imágenes, audio, video, documentos y publicaciones.

La variedad exige tecnologías flexibles de almacenamiento, procesamiento y análisis.

<!--
Notas del presentador:
Relacionar este punto con la evaluación sobre NoSQL y Data Lake. Explicar que las bases relacionales son muy útiles cuando los datos son estructurados y requieren consistencia fuerte. Sin embargo, en Big Data aparecen datos flexibles, variables y heterogéneos. Por eso surgen bases NoSQL, Data Lakes, motores distribuidos y herramientas de procesamiento que permiten trabajar con diferentes formatos sin exigir una estructura rígida desde el inicio.
-->

::leftTitle::
Bases SQL

::rightTitle::
Bases NoSQL

::left::
Organizan datos en tablas.

Usan esquemas definidos.

Favorecen relaciones, integridad y consultas estructuradas.

Son comunes en sistemas transaccionales y administrativos.

::right::
Permiten modelos flexibles.

Pueden usar documentos, columnas, grafos o clave-valor.

Facilitan escalabilidad horizontal.

Son útiles para datos variables, distribuidos o de gran volumen.

<!--
Notas del presentador:
Explicar que SQL y NoSQL no deben verse como enemigos, sino como enfoques adecuados para necesidades distintas. SQL es fuerte para transacciones, integridad y relaciones complejas. NoSQL ofrece flexibilidad y escalabilidad para ciertos escenarios de Big Data. Relacionar con la evaluación: Apache Cassandra utiliza un modelo orientado a columnas; otras bases NoSQL pueden ser documentales, clave-valor o de grafos. Destacar que elegir una base depende del problema, no de la moda tecnológica.

::title::
Data Lake y Data Warehouse

::content::
Un **Data Lake** almacena grandes volúmenes de datos en su formato original.

Un **Data Warehouse** organiza datos procesados y estructurados para análisis empresarial.

El Data Lake favorece flexibilidad y bajo costo de almacenamiento.

El Data Warehouse favorece consistencia, consulta analítica y reportes organizacionales.

<!--
Notas del presentador:
Aclarar que un Data Lake permite conservar datos crudos, estructurados, semiestructurados y no estructurados para análisis futuros. Esto es útil cuando todavía no se sabe exactamente qué preguntas se formularán. El Data Warehouse, en cambio, suele trabajar con datos transformados y modelados para indicadores de negocio. Relacionar con la evaluación: una ventaja frecuente del Data Lake es su escalabilidad y menor costo relativo de almacenamiento frente a soluciones tradicionales, aunque requiere gobierno para no convertirse en un “pantano de datos”.
-->

::title::
Pipeline de datos: de la fuente al hallazgo

::left::
**1. Fuente**

Sensores, aplicaciones, formularios, sistemas transaccionales, redes sociales, logs.

**2. Ingesta**

Captura por lotes o en tiempo real.

**3. Almacenamiento**

Data Lake, Data Warehouse, bases NoSQL o almacenamiento en la nube.

::right::
**4. Procesamiento**

Limpieza, transformación, agregación y enriquecimiento.

**5. Análisis**

Consultas, modelos, segmentación, predicción o detección de anomalías.

**6. Comunicación**

Dashboards, reportes, alertas, visualizaciones y recomendaciones.

<!--
Notas del presentador:
Presentar el pipeline como una cadena de valor. Los datos nacen en una fuente, se capturan, se almacenan, se procesan, se analizan y finalmente se comunican. Explicar que cada etapa tiene herramientas específicas y riesgos propios. Por ejemplo, una mala ingesta produce datos incompletos; un almacenamiento sin gobierno dificulta encontrar información; un análisis sin contexto puede generar conclusiones equivocadas; una visualización confusa puede afectar la toma de decisiones.
-->

::title::
Procesamiento por lotes y procesamiento en streaming

::image::
<img src="/imagenes/procesamiento-por-lotes-y-procesamiento-en-streaming.png" alt="Imagen de apoyo sobre Big Data" />

::content::
**Procesamiento por lotes**

Analiza datos acumulados en periodos definidos.

Ejemplo: reporte diario de ventas.

**Procesamiento en streaming**

Analiza datos continuos con baja latencia.

Ejemplo: detección de fraude en tiempo real o monitoreo de sensores industriales.

<!--
Notas del presentador:
Explicar que el procesamiento por lotes es útil cuando no se requiere una respuesta inmediata. En cambio, el streaming es necesario cuando el tiempo de respuesta es crítico. Relacionar con herramientas de la evaluación: Spark puede trabajar con grandes volúmenes y cuenta con Spark Streaming para flujos continuos; Apache Flink utiliza mecanismos como checkpoints y snapshots para mantener consistencia y recuperación ante fallos; Apache Storm trabaja con topologías de procesamiento; Kafka se usa para mensajería distribuida, particionamiento y durabilidad de eventos.
-->

::title::
Herramientas clave del ecosistema Big Data

::left::
**Hadoop**

Procesamiento distribuido a gran escala.

**HDFS**

Sistema de archivos distribuido.

**YARN**

Gestión de recursos en clúster.

**Spark**

Procesamiento rápido, frecuente en memoria.

::right::
**Kafka**

Mensajería distribuida y eventos.

**Flink**

Procesamiento de flujos con consistencia.

**Airflow**

Orquestación y monitoreo de flujos de trabajo.

**Presto**

Motor de consulta distribuida de baja latencia.

<!--
Notas del presentador:
Esta diapositiva conecta directamente con varias preguntas de evaluación. Aclarar que Hadoop es un marco amplio; HDFS almacena datos de forma distribuida; YARN coordina recursos del clúster; Spark permite procesar datos con alto rendimiento y puede programarse en lenguajes como Python, Java, Scala y R, pero no Ruby como lenguaje principal de su API estándar. Kafka se distingue por durabilidad y particionamiento de mensajes. Airflow no almacena ni procesa datos por sí mismo como motor principal, sino que orquesta tareas y monitorea flujos de trabajo.
-->

::title::
Cloud computing y Big Data

::content::
La nube facilita el trabajo con Big Data porque permite aprovisionar recursos bajo demanda, escalar horizontalmente, almacenar datos masivos y ejecutar servicios especializados.

Ejemplos de servicios asociados:

Amazon EC2, Auto Scaling, S3, Athena, RDS, MediaConvert, Lambda y servicios administrados para análisis de datos.

<!--
Notas del presentador:
Explicar que cloud computing permite evitar grandes inversiones iniciales en infraestructura física. La escalabilidad horizontal consiste en añadir más instancias o nodos según la demanda. Relacionar con la evaluación: EC2 permite crear servidores virtuales; Auto Scaling ajusta automáticamente la cantidad de instancias; Athena permite analizar datos almacenados en S3 mediante SQL sin administrar servidores; Lambda representa computación sin servidor; RDS es un servicio de bases de datos relacionales; MediaConvert se usa para transcodificación de video en AWS.
-->

Gobernanza, seguridad y ética de los datos

- Privacidad y protección de datos personales.

- Control de acceso y autorización.

- Calidad, trazabilidad y linaje de datos.

- Reducción de sesgos.

- Uso legítimo, transparente y proporcional de la información.

::title::
Cómo comunicar hallazgos de Big Data

::content::
Un buen hallazgo debe responder:

- ¿Qué patrón se observa?

- ¿Qué evidencia lo respalda?

- ¿Qué impacto tiene?

- ¿Qué decisión sugiere?

- ¿Qué limitación debe considerarse?

La comunicación debe ser clara, visual, breve y orientada a la acción.

<!--
Notas del presentador:
Explicar que en Big Data no basta con procesar datos; también se deben comunicar resultados de forma comprensible para quienes toman decisiones. Un dashboard saturado o una gráfica mal escogida puede ocultar el hallazgo. Recomendar títulos interpretativos, unidades claras, segmentaciones útiles y mensajes breves. Por ejemplo, no decir solo “gráfico de ingresos”, sino “el ingreso entre 6:30 a. m. y 7:30 a. m. concentra el mayor nivel de congestión”.
